Why drop better-auth?

The Modulith backend is the auth system. It issues JWTs, rotates refresh tokens, handles Google OAuth, and enforces rate limits. better-auth would duplicate or conflict with all of this. A thin BFF proxy layer is all that's needed.

Why drop oRPC?

oRPC assumes TypeScript on both ends sharing a router definition. The backend is .NET — the type bridge is the OpenAPI spec. @hey-api/openapi-ts generates the same type safety from the actual source of truth.

The .NET backend requires zero changes. The BFF is purely a frontend concern — it receives JWTs from the backend and stores them server-side, then attaches them when proxying requests.

Key Principles

  • Tokens never reach the browser. Access + refresh tokens live in an encrypted httpOnly cookie managed by Next.js.
  • Single proxy chokepoint. All API calls go through /api/proxy/[...path], which handles token attachment and silent refresh.
  • Types generated, not hand-written. The OpenAPI spec is the single source of truth for all request/response types, Zod schemas, and React Query hooks.
  • Errors are structured. The backend returns ProblemDetails consistently — a single mapper converts them to form errors or toast notifications.
  • Real-time via SSE. Bell notifications stream through GET /v1/me/notifications/stream (Server-Sent Events). The BFF proxies the SSE connection, attaching the Bearer token. The client uses an EventSource (or fetch-based reader) to receive notification.created events and invalidate React Query caches.

Workflow (separate repos)

The spec file openapi.json is committed in the frontend repo — it's the contract snapshot documenting which API version the frontend was built against.

Two scripts:

  • npm run api:sync — fetches http://localhost:5000/openapi/v1.json from the running backend → writes ./openapi.json
  • npm run api:generate — runs @hey-api/openapi-ts against the local ./openapi.json → outputs to src/api/generated/

Development: When backend endpoints change, run api:sync then api:generate, review the diff, commit.

CI: Runs api:generate against the committed spec. Frontend builds are fully decoupled from the backend being up. Type mismatches between generated code and components fail the build.

Validation chain: Backend FluentValidation rules → OpenAPI spec constraints → Generated Zod schemas → TanStack Forms. One source of truth, no manual sync.

4. BFF Auth Layer

Cookie session structure

Design constraint: The JWT does not carry permissions (the permissions array could overflow cookie size limits). Permissions are fetched separately via GET /v1/users/me after sign-in and cached by React Query.
// Sealed by iron-session into a single httpOnly cookie // Deliberately thin — no permissions, no display name { accessToken: "eyJhbG...", // JWT from backend refreshToken: "d4f8a1...", // opaque token from backend expiresAt: 1715270400, // unix timestamp (from JWT exp claim) user: { id: "guid", // from JWT sub claim email: "user@example.com", // from JWT claim role: "Admin" // from JWT claim — coarse gate only } }

Permission layer (React Query, not cookie)

Permissions live in React Query, not in the cookie. This keeps the cookie small and avoids the 4KB browser limit regardless of how many permissions the system grows into.

Middleware (coarse route protection only)

Next.js middleware runs on every navigation. It reads the session cookie (checking existence and expiry only — no decryption needed for the fast path). It does not check permissions — those are enforced client-side by React Query + <Can>.

  • No cookie or expired → redirect to /login (except public routes)
  • Valid cookie, onboarding incomplete → redirect to /onboarding
  • Admin routes → optionally check role from cookie for a coarse gate (but fine-grained permission checks happen client-side after /v1/users/me fetch)

SSE proxy considerations bell-notifications

The SSE stream (GET /v1/me/notifications/stream) passes through the catch-all proxy but requires special handling: the proxy must stream the response without buffering (no Content-Length, chunked transfer), keep the connection alive, and handle client disconnect cleanly. The proxy attaches the Bearer token like any other request, but the response is a long-lived text/event-stream that the proxy forwards chunk by chunk.

Two-Factor Authentication two-factor

Login responses now carry a discriminator. { status: "Authenticated", session } means the cookie can be sealed immediately. { status: "TwoFactorRequired", challenge: { challengeToken, expiresAt } } means the user must complete a second step at POST /v1/users/login/2fa. The BFF must not seal a cookie until the challenge is completed — the half-authenticated user has no session. The same shape applies to /v1/users/auth/google/login.

8. Phased Implementation

Phase 1 Foundation + Auth

Everything depends on this. No feature works without auth and the codegen pipeline.

  • Project scaffolding (Next.js, Tailwind v4, shadcn/ui setup)
  • OpenAPI codegen pipeline (@hey-api/openapi-ts config + npm scripts)
  • BFF auth routes (login, register, logout, refresh, session)
  • iron-session cookie management
  • Next.js middleware (auth redirect, public route allowlist)
  • Auth context provider (useCurrentUser, isAuthenticated, hasPermission)
  • ProblemDetails → form/toast error mapper
  • Login and register pages with TanStack Forms + generated Zod schemas
  • Registration mode awareness — register page adapts to Open / InviteOnly / Disabled
  • Invite-token handling: /register?token=… pre-fills email, sends token with registration
  • Graceful states for expired/invalid tokens, invite-required, and registration-disabled
  • Basic app shell (layout, navigation, user menu, bell icon with unread badge)
  • SSE connection infrastructure — EventSource to /api/proxy/v1/me/notifications/stream, reconnect logic, query invalidation on events

Exit criteria: User can register, log in, see their name in the nav, log out. Bell icon shows unread count. SSE connection streams events. Registration adapts to all three modes. Invite-link registration works end to end. Errors display correctly in forms. Expired tokens refresh silently.

Phase 2 Onboarding + Password Flows

Completes the full auth lifecycle. After this phase, the template handles every auth scenario.

  • Onboarding wizard (terms acceptance → optional initial password → complete)
  • Middleware onboarding gate (redirect to /onboarding if incomplete)
  • Google OAuth flow (login → pending confirmation → cookie seal). In InviteOnly mode, confirm step requires invitation token.
  • Forgot password page → "check email" interstitial
  • Reset password page (token from URL, new password form)
  • Google OAuth BFF routes

Exit criteria: New users complete onboarding before seeing the app. Google sign-in works end to end. Password reset via email works.

Phase 2.5 Two-Factor Authentication two-factor

Optional second factor (TOTP) + recovery codes. Lives between auth lifecycle and self-service settings because the login-challenge branch must work before any 2FA-enabled user can reach the management UI.

  • Update /api/auth/login and /api/auth/google/login BFF routes to branch on the new { status, session?, challenge? } response shape. Never seal a cookie for TwoFactorRequired.
  • New BFF route /api/auth/login/two-factor → forwards to /v1/users/login/2fa, seals cookie on success.
  • New page /login/two-factor — 6-digit code input with autocomplete="one-time-code", "Use a recovery code instead" mode, countdown to challenge expiry.
  • Auth provider login() / googleLogin() return a discriminated union; TwoFactorRequired stores the challenge token in sessionStorage and navigates to the challenge page.
  • New settings page /settings/security — three-state panel (off / setup wizard / on) with QR code render from otpAuthUri, copyable fallback secret, recovery code display, regenerate + disable flows.
  • Recovery codes: shown exactly twice (after setup, after regenerate), with copy + download + "I saved my codes" confirmation gate. Never stored client-side beyond the displaying component.
  • Codegen pickup of new types (LoginTwoFactorRequest/Response, SetupTotpResponse, ConfirmTotp*, DisableTwoFactor*, RegenerateRecoveryCodes*) via pnpm api:generate.

The Security panel reads twoFactorEnabled from GET /v1/users/me (cached in useCurrentUser); setup / disable / regenerate invalidate currentUserQueryKey to re-render.

Exit criteria: Email/password and Google sign-in both complete through the challenge page when 2FA is enabled. Recovery codes work as an alternate sign-in input. Setup wizard, regenerate, and disable flows are all functional. Expired/missing challenge tokens fall back to /login cleanly.

Phase 3 Profile + Account Management

Self-service account features. Form-heavy, exercises the error mapper and two-phase flows.

  • Profile settings page (view current user info)
  • Change password form
  • Email change: request → pending state → confirmation page
  • Google account linking / unlinking
  • Bell notification dropdown (popover from nav icon — list recent, mark read, link to full page)
  • Full notification list page (/notifications) — cursor-paginated, filter by status (unread/archived), archive action
  • Mark as read (single + mark all), archive individual notifications
  • Real-time toast on SSE notification.created event
  • Notification preferences page (/settings/notifications) — per-category bell/email toggles, locked Account/Security rows
  • GDPR: export personal data (download JSON)
  • GDPR: delete account (type-email-to-confirm dialog, post-deletion page)

Exit criteria: Users can fully self-manage their account. Bell notifications display in real-time, can be read/archived. Notification preferences save correctly with locked categories. Email change and account deletion work end to end.

Phase 4 Admin + Audit

Permission-gated admin features. Introduces TanStack Table and nuqs for real.

  • Permission-aware route guards and <Can> component
  • Admin navigation section (conditionally rendered)
  • User list page (TanStack Table, pagination via nuqs)
  • User detail page
  • Role change action (confirmation dialog)
  • Invitation management page — send invitations, view pending, revoke (requires users.invitations.write)
  • Audit trail — own activity view (regular users)
  • Audit trail — admin view (filter by user, event type)
  • Expandable JSON payload viewer

Exit criteria: Admin can manage users, send/revoke invitations, and view any audit trail. Regular users see their own activity. All admin UI hidden from non-admins.

9. Open Decisions

Resolved Separate repo
The frontend lives in its own repository. The backend template is complete on its own — consumers may not want a frontend, so it stays optional. The OpenAPI spec file is the contract bridge between the two repos.
Resolved OpenAPI spec: committed file + sync script
openapi.json is committed in the frontend repo as the contract snapshot. npm run api:sync fetches from the running backend, npm run api:generate runs codegen against the local file. CI builds are fully decoupled from the backend.
Resolved Google OAuth: GSI button (Option A)
Google Sign-In button (GSI) in the browser obtains the ID token, sends it to the BFF, which forwards it to the backend. Simpler than a full OAuth redirect flow and matches the backend's expectation (it verifies Google ID tokens directly).
Resolved Session cookie: single cookie, no permissions
Permissions are not stored in the JWT or cookie. The cookie holds only tokens + {id, email, role} — well under 4KB. Permissions are fetched via GET /v1/users/me and cached in React Query client-side. Single sealed cookie wins.
Resolved Proxy: catch-all first
Start with a single /api/proxy/[...path] catch-all. Extract specific routes only when a feature needs custom proxy logic (e.g., cache headers, response transforms). Zero maintenance when backend adds endpoints.
Resolved CORS: backend responsibility
The .NET API configures CORS to allow the Next.js BFF server's origin. This is a backend-repo task — the BFF makes server-to-server requests, so CORS must allow that origin. The browser never talks directly to the .NET API.
Resolved Dev experience: two terminals
Aspire can't manage Next.js across separate repos — AddNpmApp expects projects in the same solution tree. Two terminals: dotnet run --project src/AppHost (backend + infra) and npm run dev (Next.js). A Makefile or Taskfile can wrap both into make dev for convenience.
